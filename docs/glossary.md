# Aestus — Glossary

Term definitions used consistently across `docs/specs/cockpit_spec.md`, `docs/specs/cockpit_ui_implementation.md`, code, and agent tasks. When a term here conflicts with informal usage elsewhere, this definition wins.

---

## Asset

A financial instrument tracked by the system. Assets have a canonical identity (`canonical_id`, `symbol`, `base`, `quote`, `asset_class`) and can be listed on multiple venues under different instrument IDs. Examples: BTC perpetual, ETH/USDT spot, SPX index, DXY, GOLD, VIX.

## Venue

An exchange or data source where an asset is traded or priced. Each venue has a `venue_id` and `market_type` (perp, spot, index, macro proxy). One canonical asset may have multiple `VenueInstrument` records — one per venue where it trades. Examples: Binance (perp), Bybit (perp), Hyperliquid, OKX, TradingView (index proxy).

## Market State

The current real-time snapshot of conditions for an asset or the market as a whole — price, mark price, funding rate, open interest, 24-hour volume, returns across timeframes, volatility regime, breadth, and data freshness. Market state is deterministic and comes from ingestion + feature engine outputs, never from the LLM.

## Feature Snapshot

A versioned, timestamped record of computed rolling features for an asset at a specific point in time. Includes: returns (1m/5m/15m/1h/24h), realized volatility, z-scores, funding z-score, OI delta, volume z-score, correlation set, cross-venue basis, and regime labels. Produced by the Rust feature engine; consumed by anomaly detection and the context packet builder.

## Anomaly

A detected unusual market condition that crosses a configured threshold or statistical deviation. Anomaly types: `funding_spike`, `oi_surge`, `volume_anomaly`, `correlation_break`, `basis_dislocation`, `liquidation_cluster`, `whale_flow`, `macro_approaching`, `news_cluster`. Each anomaly has a severity, sigma value, affected assets/venues, status lifecycle (active → snoozed / resolved / expired / dismissed), and links to the context it triggered.

## Context Packet

A structured, persisted assembly of data built around a trigger anomaly before the LLM sees it. Contains: the trigger anomaly, market state snapshot for the affected asset, correlated asset state (e.g., ETH, SPX, DXY, GOLD, VIX), recent relevant news, upcoming macro events, on-chain context, historical analogues if available, and deterministic levels from the level engine. The packet is the sole input to briefing generation. Source freshness metadata is always included.

## Briefing

A proposal generated by the LLM from a context packet. Always has a `stance`: `long`, `short`, or `no_trade`. Contains: thesis, key supporting/contradicting factors, confidence with reasoning, deterministic entry zone / invalidation / targets (provided by the level engine — the LLM may select among them and explain, but never invent new values), re-check conditions, and cost/model metadata. Briefings are decision support, not commands. `no_trade` is a first-class and frequently correct outcome.

## Decision

A logged user response to a briefing. One of: `act`, `skip`, `snooze`, `dismiss`, `watch`. Recorded with: the briefing that triggered it, the user's rationale, any planned entry / stop / target levels, risk parameters (if acting), and the timestamp. Every decision is persisted — it is the primary input to the learning loop and analytics.

## Setup

A repeating configuration of market conditions (anomaly type + regime + feature profile) that the user believes may offer edge. Setups are named (e.g., "BTC funding flush + OI squeeze"), documented in Playbooks, tagged on journal entries, and broken out in analytics to measure actual edge over time.

## Regime

A labeled, deterministically computed market state used to contextualize anomalies and analytics. Examples: `risk_on` / `risk_off`, `low_vol` / `med_vol` / `high_vol`, `trending` / `ranging`. Regimes are derived from feature snapshots (realized vol, breadth, correlation state) and are never assigned by the LLM. Regime context appears in briefings as input, not as LLM opinion.

## R-Multiple (R)

A dimensionless measure of trade outcome relative to planned initial risk. `R = (exit_price − entry_price) / (entry_price − stop_price)` for longs (adjusted sign for shorts). R = 1.0 means the trade returned exactly the amount risked. R = −1.0 means a full stop loss. Used throughout journal and analytics for risk-normalized performance comparison across setups, regimes, and timeframes. Sizing guidance in briefings is also expressed in R terms (e.g., "0.5R position").

## Invalidation

The price level or condition at which a trade thesis is considered wrong and the position or idea should be exited or abandoned. Invalidation levels are always produced by the deterministic level engine (ATR bands, swing structure, volatility clusters) and are never invented by the LLM. Every directional briefing must include an explicit invalidation with its derivation source.
